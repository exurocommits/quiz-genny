# Agentic Quiz Generator — Complete Technical Specification v1.0

## Document Purpose

This document is the authoritative specification for building an AI-assisted quiz generation tool. It is structured for handoff to an autonomous development agent (Claude Code) to assume full implementation responsibility. Every feature, user interaction, system behavior, data structure, and edge case is documented. The developer should be able to build the complete application using only this document and standard technical judgment.

---

# PART 1: PRODUCT DEFINITION

## 1.1 Product Overview

### Core Concept

An agentic workflow tool that assists quiz hosts in generating professional, balanced, and verified quiz content. The system uses LLM-powered generation with human-in-the-loop curation at each stage. Final output is a downloadable PowerPoint presentation ready for live hosting.

### Key Principles

1. **AI generates, human curates** — AI produces candidates in batches, user selects and refines through intuitive accept/reject actions
2. **Difficulty balancing** — Questions are scored across four dimensions to ensure engaging score distributions that keep all participants engaged
3. **Verification by default** — All answers are fact-checked via web search before final output, with user confirmation on corrections
4. **Progressive refinement** — Each "Generate more" action incorporates prior feedback, rejection memory, and difficulty gap awareness
5. **Duplicate prevention** — System actively prevents duplicate or highly similar questions across the entire quiz
6. **Professional output** — Final PPTX is presentation-ready with consistent theming, curated imagery, and optional host notes

### Target Users

Quiz hosts creating content for:
- Pub quizzes
- Corporate team-building events
- Charity fundraiser nights
- Competitive quiz leagues
- Social gatherings and parties
- Educational assessments
- Family game nights

### Technical Approach

- Web application (React frontend)
- Node.js or Python backend
- LLM integration via Anthropic API (Claude)
- Web search integration for verification and image sourcing
- PowerPoint generation via server-side library
- Session-based state management (no user accounts in v1)

### 2.1.6 Difficulty Distribution

**Purpose:** Sets the target mix of easy, medium, and hard questions per round to ensure balanced scoring.

**Background — Difficulty Scoring System:**

Each question is scored on four dimensions (1-5 scale):

| Dimension | What It Measures |
|-----------|------------------|
| Knowledge Accessibility | How widely known is this fact? |
| Cognitive Load | How much thinking/reasoning required? |
| Cultural/Linguistic Accessibility | Is it globally accessible or culturally specific? |
| Temporal Relevance | Is it timeless or dependent on recent events? |

**Overall Difficulty Calculation:**
```
Overall = (Knowledge + Cognitive + Cultural + Temporal) / 4
```

**Banding:**
| Score Range | Band |
|-------------|------|
| 1.0 – 2.0 | Easy |
| 2.1 – 3.2 | Medium |
| 3.3 – 5.0 | Hard |

**UI Component:** Three sliders or preset selector.

**Preset Options:**

| Preset | Easy | Medium | Hard | Use Case |
|--------|------|--------|------|----------|
| Accessible | 6 | 3 | 1 | Social, charity, family |
| Balanced | 4 | 4 | 2 | General pub quiz |
| Challenging | 2 | 4 | 4 | League, competitive |
| Custom | — | — | — | Manual sliders |

**Slider Behavior (Custom mode):**
- Three linked sliders that must sum to questions per round
- Adjusting one automatically adjusts others proportionally
- Or: two sliders, third is calculated remainder
- Visual stacked bar shows proportion

**Default:** Balanced

**Behavior:** Target distribution stored in state. Used during question generation to guide output, and during curation to show progress toward target.

## Phase 2: Section/Round Theme Selection

### 2.2.1 Screen Layout

**Header:** Quiz summary bar (sticky) showing "5 rounds × 10 questions = 50 total | Balanced difficulty | Casual tone"

**Main area:** Grid of section cards (responsive: 3-4 columns desktop, 2 columns tablet, 1 column mobile)

**Footer:** Action bar with selection counter, "Generate More" button, and "Continue" button

### 2.2.2 Initial Section Generation

**Trigger:** Automatic on entering Phase 2

**System Action:**

```
LLM Call: Generate Section Ideas
├── Model: Claude (claude-sonnet-4-20250514 recommended for speed/cost balance)
├── System prompt: Include audience preset, selected attribute tags, tone, context notes
├── User prompt: "Generate 30 unique quiz round theme ideas suitable for this audience..."
├── Output format: Structured JSON array
├── Response structure per section:
│   {
│     "id": "uuid",
│     "name": "90s Pop Music",
│     "description": "Chart-toppers, one-hit wonders, and iconic albums from the 1990s",
│     "estimated_difficulty": "tends_easy",
│     "category": "entertainment"
│   }
└── Post-processing: Validate JSON, assign UUIDs if not present, store in state
```

**Estimated Difficulty Logic:**
- LLM assesses based on theme + audience calibration
- Values: "tends_easy", "tends_medium", "tends_hard", "mixed"
- This is an estimate only; actual difficulty determined at question level

**Category Assignment:**
- LLM assigns broad category for optional filtering
- Categories: Entertainment, Science & Nature, History, Geography, Sports, Arts & Literature, Food & Drink, Music, Film & TV, General Knowledge, Lifestyle, Technology, Potpourri

### 2.2.3 Section Card Display

**Card Contents:**
- Section name (prominent)
- Description (2-3 sentences)
- Estimated difficulty badge (color-coded: green=easy, amber=medium, red=hard, grey=mixed)
- Category tag (small, subtle)
- Selection state indicator (checkbox or visual highlight)

**Card Interactions:**
- Click anywhere on card to toggle selection
- Selected cards: elevated shadow, colored border, checkmark overlay
- Hover state: slight lift, cursor pointer

### 2.2.4 Selection Counter and Validation

**UI Component:** Sticky footer bar

**Display:** "3 of 5 rounds selected"

**Visual:** Progress dots or segments filling as selections made

**Behavior:**
- Counter updates immediately on selection/deselection
- Cannot select more than required rounds
- If user tries to select beyond limit: either prevent click or auto-deselect oldest selection (configure: prevent is safer UX)

### 2.2.5 "Surprise Me" Auto-Select

**Purpose:** Quick path for users who want AI to choose a balanced set of sections.

**UI Component:** Secondary button in footer bar, "Surprise Me"

**Behavior on click:**
1. Clear any current selections
2. AI selects optimal combination based on:
   - Variety of categories (avoid clustering)
   - Mix of estimated difficulties
   - Relevance to audience attributes
3. Animate selections appearing (sequential highlight, 200ms each)
4. User can still manually adjust after

**Implementation:**
```
LLM Call: Select optimal sections
├── Input: All 30 generated sections, target round count, audience context
├── Prompt: "Select [X] sections that would create the most engaging and varied quiz for this audience..."
├── Output: Array of section IDs
└── Action: Set selected state for returned IDs
```

**Alternative simpler implementation:**
```
Algorithm: Weighted random selection
├── Score each section by:
│   - Category diversity bonus (avoid repeats)
│   - Difficulty variety bonus (mix easy/hard)
│   - Audience relevance score (keyword match to tags)
├── Select top-scoring combination
└── With some randomness to avoid deterministic output
```

### 2.2.6 "Generate More" Sections

**Purpose:** If user doesn't like available options, generate additional section ideas.

**UI Component:** 
- Secondary button "Generate More"
- Expandable/collapsible text input below: "Optional: Guide the next generation"

**Behavior on click:**
1. Show loading state on button
2. Make LLM call for 15 more sections (not 30, to avoid overwhelming)
3. New sections appended to grid (not replacing)
4. Smooth scroll/highlight to indicate new content
5. Previously generated sections remain (including selected ones)

**LLM Call includes:**
- All previously generated section names (to avoid duplicates)
- Any user feedback from text input
- Original audience context

**Prompt structure:**
```
"Generate 15 more unique quiz round theme ideas. 
Avoid these already-generated themes: [list of existing section names]
User feedback for this generation: [user input or 'None provided']
Audience context: [preset + tags + tone + notes]"
```

**Generation limit:** Soft limit of 5 "Generate More" clicks (75 additional sections). After limit: show message "You've generated a lot of options! If none work, try adjusting your audience settings."

### 2.2.7 Section Reordering

**Purpose:** Set the order rounds will appear in the final quiz.

**UI:** Once X sections are selected, show a secondary panel or modal "Set Round Order"

**Component:** Drag-and-drop sortable list of selected sections

**Behavior:**
- Default order: order of selection
- User can drag to reorder
- Order stored in state as array of section IDs

**Alternative UX:** Skip explicit reordering; order = selection order. Allow reorder in final preview phase. (Simpler to implement, may be sufficient for v1.)

### 2.2.8 Difficulty Balance Warning

**Purpose:** Warn user if selected sections skew heavily toward one difficulty.

**Trigger:** After selection count reaches target

**Logic:**
```
Count estimated difficulties of selected sections:
- If ≥80% "tends_hard": show warning
- If ≥80% "tends_easy": show warning (less critical but note it)
- Otherwise: no warning
```

**UI:** Inline alert box above "Continue" button

**Message example:** "Heads up: Your selected sections tend toward harder themes. You may find it difficult to generate enough easy questions to meet your target distribution. Continue anyway or adjust your selections."

**Actions:** "Continue Anyway" (proceeds) | "Adjust Selections" (dismisses, user changes selections)

### 2.2.9 Continue to Question Generation

**UI Component:** Primary button "Continue" in footer bar

**Enabled when:** Selected section count equals required round count

**On click:**
- Store selected sections with order in state
- Clear generation memory (rejected themes) for fresh start
- Navigate to Phase 3, starting with Round 1

## Phase 3: Question & Answer Generation

### 2.3.1 Screen Layout

**Split panel layout (desktop):**
- Left panel (60%): Generation and curation area
- Right panel (40%): Accepted questions list

**Mobile:** Stacked layout with tab switching between candidates and accepted

**Header:** 
- Round indicator: "Round 2 of 5: 90s Pop Music"
- Section description
- Inspiration tags (generated alongside questions, showing themes within the round)

### 2.3.2 Initial Question Generation

**Trigger:** Automatic on entering each round's question generation screen

**System Action:**

```
LLM Call: Generate Questions
├── Model: Claude (claude-sonnet-4-20250514)
├── System prompt: Difficulty calibration, tone, avoid duplicates instruction
├── User prompt: "Generate 15 quiz questions for the round theme '[Section Name]'..."
├── Required output structure (JSON):
│   {
│     "inspiration_tags": ["British artists", "one-hit wonders", "music videos"],
│     "questions": [
│       {
│         "id": "uuid",
│         "question": "Which British band had a 1997 hit with 'Bittersweet Symphony'?",
│         "answer": "The Verve",
│         "difficulty_scores": {
│           "knowledge_accessibility": 2,
│           "cognitive_load": 1,
│           "cultural_linguistic": 3,
│           "temporal_relevance": 2
│         },
│         "overall_difficulty": 2.0,
│         "difficulty_band": "easy",
│         "reasoning": "Well-known song, frequently referenced in media, straightforward recall"
│       }
│     ]
│   }
└── Post-processing: Validate JSON, run duplicate detection, store in state
```

**Duplicate Detection Integration:**

Before storing generated questions, check each against:
1. All previously accepted questions in other rounds
2. All previously generated (not yet rejected) questions in current session

**Duplicate detection logic:**
```
For each new question:
├── Semantic similarity check against existing questions
│   ├── Method: Embedding comparison OR
│   ├── Method: LLM call "Are these questions asking essentially the same thing?"
│   └── Threshold: >0.85 similarity = duplicate
├── Answer overlap check
│   └── If same answer AND similar topic = likely duplicate
└── If duplicate detected:
    ├── Flag question with "duplicate_of": [existing question ID]
    └── Show badge in UI, allow user to keep if desired (edge case: same answer, different question is valid)
```

**Note on LLM repeat penalties:** LLMs have temperature and frequency/presence penalty parameters, but these operate within a single generation, not across calls. Cross-call duplicate prevention requires explicit tracking as described above.

### 2.3.5 Live Gap Analysis

**Purpose:** Guide user toward completing a balanced round.

**Display:** Below the difficulty distribution bar

**Dynamic message examples:**
- "You need 2 more easy questions and 1 hard question"
- "Almost there! Just 1 more medium question"
- "Round complete! Review your questions or continue"

**Behavior:** Updates immediately on accept/reject actions.

### 2.3.6 "Generate More" Questions

**UI Component:**
- Button "Generate More Questions"
- Expandable text input: "Guide this generation (optional)"

**Automatic prompt enhancement:**
Based on current gaps, system constructs optimized prompt:

```
User clicked "Generate More"
├── Current gaps: Easy -2, Medium -1, Hard +1 (have extra hard)
├── Rejected question themes: ["Beatles songs", "album covers", "British artists"]
├── System constructs prompt:
│   "Generate 10 more questions for '90s Pop Music'.
│    Prioritize: 2 EASY questions, 1 MEDIUM question
│    Avoid themes already covered or rejected: Beatles songs, album covers, British artists
│    User guidance: [user input if provided]"
└── Append results to candidates list
```

**Generation limit:** 5 "Generate More" per round. After limit: "Having trouble finding the right questions? Try adjusting this round's theme or difficulty targets."

### 2.3.7 Duplicate Detection Badge Details

**When displayed:** Question is flagged as similar to existing accepted question

**Badge appearance:** Yellow/orange warning pill "Similar to existing"

**Click interaction:** Tooltip or modal showing:
- The original question it's similar to
- Which round the original is in
- Options: "Keep anyway" | "Reject"

**"Keep anyway" behavior:**
- User confirms they want both questions despite similarity
- Badge changes to grey "Acknowledged duplicate"
- Both questions remain

### 2.3.8 Round Completion

**Trigger:** Accepted question count equals target AND difficulty distribution within tolerance

**Tolerance:** Allow ±1 per difficulty band (e.g., target 4 easy, accept 3-5)

**UI:** 
- Distribution bar shows "Complete" state (all green)
- "Next Round" button becomes primary
- Optional: celebration micro-animation

**If imbalanced completion:**
- User has 10 questions but wrong distribution
- Warning: "Your difficulty balance is off (3 easy, 5 medium, 2 hard vs. target 4/4/2). Continue anyway?"
- "Continue with imbalance" | "Adjust questions"

### 2.3.9 Navigation Between Rounds

**After completing round 1:** "Next Round" goes to round 2's question generation

**Round indicator in header:** Clickable, allows jumping back to previous rounds to make edits

**State preservation:** All rounds maintain their state; returning to a completed round shows the accepted questions with ability to modify

**Progress indicator:** Step dots or progress bar showing overall completion

### 2.3.10 Phase 3 Completion

**Trigger:** All rounds have required questions accepted

**UI:** "All rounds complete! Proceed to Verification" button

**Transition:** Navigate to Phase 4

## Phase 4: Verification

### 2.4.1 Screen Layout

**Header:** "Verifying Your Quiz" with overall progress bar

**Main area:** Three collapsible sections for result categories

**Footer:** "Continue to Images" button (disabled until all resolved)

### 2.4.2 Verification Process

**Trigger:** Automatic on entering Phase 4

**System Action:**

```
Verification Agent (parallelized across questions)
├── For each Q&A:
│   ├── Web search: Query constructed from question + answer
│   │   └── Example: "The Verve Bittersweet Symphony 1997" 
│   ├── Analyze search results
│   │   ├── Look for confirming sources
│   │   ├── Look for contradicting information
│   │   └── Assess source quality (prefer authoritative: Wikipedia, official sites, news)
│   ├── Determine status:
│   │   ├── VERIFIED: Multiple reliable sources confirm answer
│   │   ├── CORRECTED: Sources indicate different answer
│   │   └── UNVERIFIABLE: Insufficient information to confirm or deny
│   └── Store sources and confidence reasoning
└── Aggregate results, group by status
```

**Parallelization:** Run verification in batches (e.g., 10 concurrent) to balance speed and rate limits.

**Progress indication:** 
- "Verifying question 23 of 50..."
- Progress bar fills as questions complete
- Estimated time remaining

### 2.4.3 Results Display — Verified Questions

**Section header:** "✓ Verified (42 questions)" — collapsible, collapsed by default

**Content when expanded:**
- List of questions with "Verified" badge
- Each shows: question, answer, source snippet (truncated)
- Expandable to see full source details

**No action required** — these are confirmed correct

### 2.4.4 Results Display — Corrected Questions

**Section header:** "⚠ Corrections Needed (5 questions)" — expanded by default if any exist

**Card for each corrected question:**
- Question text
- Original answer (with strikethrough)
- Suggested corrected answer (highlighted)
- Source snippet showing correction evidence
- Difficulty impact warning (if correction shifts difficulty band)
- Actions: "Accept Correction" | "Keep Original" | "Edit Manually"

**Difficulty impact logic:**
```
If correction changes answer:
├── Re-run difficulty scoring on corrected Q&A
├── Compare new difficulty_band to original
├── If different: show warning
│   └── "This correction changes the difficulty from Easy to Medium"
└── Update round distribution display
```

**"Accept Correction" behavior:**
- Replaces original answer with corrected answer
- Updates difficulty scores if changed
- Moves question to Verified category

**"Keep Original" behavior:**
- User asserts original is correct (perhaps AI verification was wrong)
- Mark as "User Confirmed" — not AI verified
- Move to Verified category with note

**"Edit Manually" behavior:**
- Opens inline edit fields for question and answer
- On save: re-runs verification on new answer (or marks as User Provided)

### 2.4.5 Results Display — Unverifiable Questions

**Section header:** "? Needs Review (3 questions)" — expanded by default if any exist

**Card for each unverifiable question:**
- Question text
- Current answer
- Reason: "Could not find reliable sources to verify this answer"
- Search queries attempted (transparency)
- Actions: "Confirm Answer is Correct" | "Edit Answer" | "Replace Question"

**"Confirm Answer is Correct" behavior:**
- User asserts correctness from their knowledge
- Mark as "User Confirmed"
- Move to Verified

**"Edit Answer" behavior:**
- Inline edit field
- Optional: re-run verification after edit

**"Replace Question" behavior:**
- Opens mini-generation UI
- "Generate a replacement question for [Round Name]"
- Shows 3-5 candidates
- User selects one
- New question goes through verification before being added

### 2.4.6 Verification Completion

**Trigger:** All questions in Verified or User Confirmed state (none pending in Corrected or Unverifiable)

**UI:** 
- "All questions verified!" message
- "Continue to Images" button enabled
- Optional stats: "42 auto-verified, 5 corrected, 3 manually confirmed"

### 2.5.2 Image Style Preference

**Trigger:** Automatic on entering Phase 5

**System Action:**

```

├── For each question:
│   ├── Extract visual subject
│   │   └── LLM call: "What is the best visual subject for this quiz question: [question + answer]?"
│   │   └── Output: "The Verve band" or "Bittersweet Symphony music video" or "90s British rock"

│   │   └── LLM scoring or heuristic (size, orientation, quality indicators)

```

```

├── If <3 results, proceed to Tier 2
Tier 2: Category subject ("90s British rock band")
├── If <3 results, proceed to Tier 3
Tier 3: Abstract/thematic ("music concert stage")
├── If <3 results, proceed to Tier 4

### 2.6.2 Estimated Runtime Calculator

**Purpose:** Help host plan their event timing.

**UI Component:** Prominent display in sub-header: "Estimated runtime: ~45 minutes"

**Calculation:**

```
Base times (configurable defaults):
├── Per question (read + thinking time): 30 seconds
├── Per answer reveal: 15 seconds
├── Per round intro slide: 20 seconds
├── Buffer between rounds: 30 seconds
└── Title and closing slides: 60 seconds total

Calculation:
├── question_time = num_questions × 30
├── answer_time = num_questions × 15
├── round_overhead = num_rounds × (20 + 30)
├── bookend_time = 60
├── total_seconds = question_time + answer_time + round_overhead + bookend_time
└── Display as "~X minutes" (rounded to nearest 5)

Example (5 rounds × 10 questions = 50 questions):
├── 50 × 30 = 1500s questions
├── 50 × 15 = 750s answers  
├── 5 × 50 = 250s rounds
├── 60s bookends
├── Total = 2560s = ~43 minutes → "~45 minutes"
```

## 3.5 PPTX Generation Specification

### Slide Structure

```
Title Slide
├── Quiz title (editable)
├── Optional logo
├── Optional subtitle ("50 Questions, 5 Rounds")
└── Theme styling applied

For each round:
├── Round Intro Slide
│   ├── "Round {n}: {Section Name}"
│   ├── Section description
│   └── Optional round-specific graphic
│
├── Question Slides (×10 or configured number)
│   ├── Question number badge: "Q{n}"
│   ├── Question text (large, readable)
│   ├── Image (positioned right or bottom, if not text-only)
│   ├── Theme styling
│   └── Host notes in speaker notes field (if enabled)
│
└── Answer Slides (if reveal point)
    ├── Question text (repeated)
    ├── Answer text (revealed position, different color/style)
    ├── Same image as question slide
    └── Round/question number reference

Closing Slide
├── "Thanks for playing!" or custom text
├── Optional logo
└── Final theming
```

### Animation Approach

Given complexity concerns, use **two-slide approach** (not animations):

**Question Slide:** Shows question + image, answer NOT present
**Answer Slide:** Shows question + image + answer (answer in distinct style)

Host advances: Question → Answer → Next Question → etc.

This is more robust than click-to-reveal animations and works reliably across PowerPoint versions.

const applyStyleModifier = (query: string, style: ImageStyle): string => {

  // Tier 1: Specific subject
  const subject = await extractVisualSubject(question);
  let images = await searchAllApis(subject, style);
  
  if (images.length >= 3) {
    return { questionId: question.id, selectedImage: images[0], alternatives: images.slice(1, 6), fallbackTier: 1, isTextOnly: false };
  }
  
  // Tier 2: Category subject
  const categorySubject = await extractCategorySubject(question);
  images = await searchAllApis(categorySubject, style);
  
  if (images.length >= 3) {
    return { questionId: question.id, selectedImage: images[0], alternatives: images.slice(1, 6), fallbackTier: 2, isTextOnly: false };
  }
  
  // Tier 3: Abstract/thematic
  const thematicSubject = getThematicFallback(question);
  images = await searchAllApis(thematicSubject, style);
  
  if (images.length >= 1) {
    return { questionId: question.id, selectedImage: images[0], alternatives: images.slice(1, 6), fallbackTier: 3, isTextOnly: false };
  }
  
  // Tier 4: Text-only
  return { questionId: question.id, selectedImage: null, alternatives: [], fallbackTier: 4, isTextOnly: true };
};
```

## 3.7 Verification Implementation

### Web Search Verification

```javascript
const verifyQuestion = async (question: Question): Promise<VerificationResult> => {
  // Construct search query
  const searchQuery = `${question.answer} ${extractKeyTerms(question.question)}`;
  
  // Execute web search (using Claude's tool use or Tavily)
  const searchResults = await webSearch(searchQuery);
  
  // Analyze results with LLM
  const analysis = await analyzeVerification(question, searchResults);
  
  return {
    questionId: question.id,
    status: analysis.status, // 'verified' | 'corrected' | 'unverifiable'
    originalAnswer: question.answer,
    correctedAnswer: analysis.correctedAnswer,
    sources: analysis.sources,
    confidence: analysis.confidence,
  };
};

const analyzeVerification = async (question: Question, searchResults: SearchResult[]) => {
  const prompt = `
    Question: ${question.question}
    Provided Answer: ${question.answer}
    
    Search Results:
    ${searchResults.map(r => `Source: ${r.title}\nSnippet: ${r.snippet}`).join('\n\n')}
    
    Determine:
    1. Is the provided answer correct according to these sources?
    2. If not, what is the correct answer?
    3. How confident are you? (high/medium/low)
    
    Output JSON:
    {
      "status": "verified|corrected|unverifiable",
      "correctedAnswer": "only if corrected",
      "confidence": "high|medium|low",
      "reasoning": "brief explanation",
      "sources": [{"url": "...", "title": "...", "snippet": "relevant excerpt"}]
    }
  `;
  
  return await callClaude(prompt);
};
```

### Re-scoring After Correction

```javascript
const rescoreDifficulty = async (question: Question, newAnswer: string): Promise<Question> => {
  const updatedQuestion = { ...question, answer: newAnswer };
  
  const prompt = `
    Re-evaluate difficulty for this corrected question:
    Question: ${updatedQuestion.question}
    New Answer: ${newAnswer}
    Original Answer: ${question.answer}
    
    Score on 1-5 for each dimension...
  `;
  
  const newScores = await callClaude(prompt);
  
  const difficultyShifted = newScores.difficultyBand !== question.difficultyBand;
  
  return {
    ...updatedQuestion,
    difficultyScores: newScores.difficultyScores,
    overallDifficulty: newScores.overallDifficulty,
    difficultyBand: newScores.difficultyBand,
    verification: {
      ...updatedQuestion.verification,
      difficultyShifted,
    },
  };
};
```

## 5.1 Project Summary

You are building an **Agentic Quiz Generator** — a web application that helps quiz hosts create professional, balanced trivia quizzes with AI assistance. The tool guides users through a multi-phase workflow:

1. **Setup:** Configure audience, tone, structure, difficulty targets
2. **Sections:** AI generates round themes, user selects and orders them
3. **Questions:** AI generates Q&A pairs per round, user curates with difficulty balancing
4. **Verification:** All answers fact-checked via web search, corrections surfaced
5. **Images:** Royalty-free images selected for each question
6. **Preview/Export:** Final review, customization, and PowerPoint download

## 5.4 Critical Implementation Notes

### Duplicate Detection
LLMs don't have cross-call memory. You MUST explicitly track all accepted questions and pass them to the duplicate detection system. Use either:
- Embedding similarity (faster, requires embedding API)
- LLM comparison (more accurate, slower, use for flagged cases)

### Difficulty Scoring
Every question must be scored on four dimensions (1-5 each):
1. Knowledge Accessibility
2. Cognitive Load  
3. Cultural/Linguistic Accessibility
4. Temporal Relevance

Overall = average. Bands: 1.0-2.0 Easy, 2.1-3.2 Medium, 3.3-5.0 Hard.

Calibration shifts based on audience preset (e.g., "competitive league" has higher baseline).

### "Generate More" Context
Every regeneration call must include:
- Previously rejected question themes
- Current difficulty distribution gaps
- User feedback text (if provided)
- Existing accepted questions (for duplicate avoidance)

### PPTX Structure
Use two-slide approach (not animations):
- Question slide: Question + Image (no answer)
- Answer slide: Question + Image + Answer

This is more reliable across PowerPoint versions.

### Image Fallback Chain
1. Specific subject search
2. Category-level search
3. Abstract/thematic search
4. Text-only slide (no image)

Track which tier was used and display to user.

### Verification Confidence
Be conservative. If sources conflict or are low-quality, mark as "unverifiable" rather than making a wrong correction. User confirmation is the safety net.
